# App Design — Генератор 2D-контента для игр

## 1. Цель проекта

Создать программу, которая по короткому текстовому запросу пользователя генерирует **готовый игровой 2D-ассет в формате SVG**.

Пользователь говорит, что нужно («собака», «рабочий», «бочка», «зомби»), а система сама:

- понимает тип объекта;
- применяет встроенный минималистичный стиль;
- строит простую, но узнаваемую графику;
- сохраняет результат как SVG-файл, пригодный для игр.

Конечная цель: инструмент, с которым можно быстро наполнять игру персонажами, существами, монстрами и объектами мира **без ручного рисования и без повторного объяснения стиля**.

---

## 2. Проблема, которую решаем

Создание даже простой 2D-графики для прототипов и indie-игр занимает много времени:

- художника может не быть;
- ИИ-генераторы изображений часто дают сложный, «живописный» стиль, плохо подходящий под UI/тайловые/colony-sim игры;
- каждый раз приходится заново описывать стиль, ракурс и ограничения;
- результат часто нельзя сразу вставить в игру как чистый векторный ассет.

Нужен генератор, где стиль, правила композиции и ограничения уже «вшиты» в продукт.

---

## 3. Конечный результат (Definition of Done продукта)

Программа считается успешной, если:

1. Пользователь вводит короткий запрос на естественном языке.
2. Без дополнительных пояснений о стиле получает SVG-файл объекта.
3. Графика выглядит как простой management/colony-sim арт в духе **RimWorld** / **Prison Architect**.
4. Объект узнаваем с первого взгляда даже в маленьком размере.
5. Один и тот же стиль соблюдается для персонажей, животных, монстров и предметов.
6. Файл можно открыть в браузере/редакторе и использовать в игровом пайплайне.

Пример:

| Ввод пользователя | Ожидаемый вывод |
|---|---|
| `собака` | `dog.svg` — вид сверху/¾, простые формы, читаемый силуэт |
| `охранник` | `guard.svg` — человек в простой экипировке, тот же стиль |
| `бочка` | `barrel.svg` — объект мира, согласованный по масштабу и языку форм |
| `скелет-воин` | `skeleton_warrior.svg` — монстр/враг без смены стиля |

## 4. Целевой визуальный стиль (вшитый в продукт)

### 4.1. Референсы

Основной ориентир:

- **RimWorld** — читаемые юниты и пропы, простые формы, функциональность важнее детализации;
- **Prison Architect** — схематичные, «иконные» объекты, ясный силуэт, минимум декора.

### 4.2. Принципы стиля

Стиль должен быть зафиксирован внутри программы и применяться автоматически:

- **Ракурс:** сверху / слегка ¾ top-down (colony-sim / management look).
- **Сложность:** максимально простая геометрия (круги, овалы, прямоугольники, короткие path’ы).
- **Детализация:** низкая; достаточно признаков, чтобы объект узнавался.
- **Цвет:** плоские заливки, ограниченная палитра, без фотореализма и сложного шэйдинга.
- **Обводка:** тонкая, спокойная, не доминирующая; либо мягкое разделение форм цветом.
- **Силуэт:** главный критерий качества — объект читается даже в 32–64 px.
- **Масштаб:** персонажи, звери и предметы должны выглядеть как элементы одной игровой вселенной.
- **Без «арта ради арта»:** никакой сложной перспективы, текстур, бликов, градиентных портретов, watercolor/anime/pixel-paint эффектов.

### 4.3. Что стиль НЕ должен делать

- Не имитировать реалистичную иллюстрацию.
- Не требовать от пользователя описания «нарисуй в стиле RimWorld».
- Не использовать сложные эффекты, которые плохо переносятся в SVG и ломают консистентность.
- Не делать каждый ассет в случайном художественном направлении.

---

## 5. Основной пользовательский сценарий

1. Пользователь открывает генератор.
2. Вводит запрос: что создать.
3. (Опционально) уточняет категорию или имя файла — но это не обязательно.
4. Нажимает «Сгенерировать».
5. Система:
   - интерпретирует запрос;
   - выбирает шаблон/структуру объекта;
   - применяет встроенный стиль и правила композиции;
   - генерирует SVG;
   - показывает превью;
   - сохраняет файл.
6. Пользователь скачивает/копирует SVG и использует в игре.

Ключевое UX-правило: **стиль не спрашиваем**. Он уже часть продукта.

---

## 6. Типы контента (обязательный охват)

Генератор должен уметь создавать:

### 6.1. Персонажи
Люди и гуманоидные роли: рабочий, охранник, врач, поселенец, солдат и т.п.

### 6.2. Животные / звери
Собака, корова, волк, птица, лошадь и другие узнаваемые животные.

### 6.3. Монстры / враги
Зомби, скелет, слизь, демон, мутант и другие простые фантазийные существа.

### 6.4. Объекты мира
Мебель, контейнеры, инструменты, ресурсы, декорации: бочка, ящик, стол, кровать, костёр, дерево, камень и т.д.

Все категории рисуются **одним визуальным языком**.

---

## 7. Функциональные требования

### FR-1. Текстовый ввод
Система принимает короткий запрос на естественном языке (русский и/или английский).

### FR-2. Автоматическое определение типа ассета
По запросу система классифицирует объект:
- character
- animal
- monster
- prop / world object

### FR-3. Встроенный style engine
Перед генерацией всегда применяются фиксированные правила стиля colony-sim / management art. Пользователь не обязан их указывать.

### FR-4. Генерация SVG
Результат — валидный `.svg` файл с прозрачным фоном.

### FR-5. Превью
После генерации показывается визуальное превью ассета.

### FR-6. Сохранение / экспорт
Пользователь может сохранить файл с осмысленным именем (`dog.svg`, `barrel.svg` и т.д.).

### FR-7. Повторяемость стиля
Разные запросы дают визуально согласованные результаты (одинаковый язык форм, толщина линий, подход к цвету и масштабу).

### FR-8. Узнаваемость
Даже при простой геометрии объект должен быть однозначно распознаваем.

### FR-9. Устойчивость к коротким запросам
Запрос вида «собака» уже достаточен. Система сама дополняет недостающие параметры стиля и композиции.

### FR-10. Внутренняя база правил качества
В логике программы должны быть заложены:
- правила силуэта;
- правила пропорций;
- ограничения на число деталей;
- правила палитры;
- шаблоны строения для типовых классов объектов.

---

## 8. Нефункциональные требования

### NFR-1. Простота для ИИ-пайплайна
Формат и стиль должны быть достаточно простыми, чтобы LLM / генеративная логика стабильно справлялась с задачей.

### NFR-2. Детерминированность стиля
Стиль не «плавает» от запроса к запросу.

### NFR-3. Пригодность для игр
SVG должен быть чистым, без лишнего мусора, удобным для импорта/конвертации.

### NFR-4. Читаемость на игровом масштабе
Ассет должен оставаться понятным в маленьком размере.

### NFR-5. Расширяемость
Архитектура должна позволять позже добавить:
- вариации одного объекта;
- наборы ассетов;
- анимационные кадры;
- экспорт в PNG/spritesheet;
- пакетную генерацию.

### NFR-6. Локальный приоритет понятности
Лучше простой и узнаваемый результат, чем сложный и «красивый», но нечитаемый.

---

## 9. Требования к качеству графики

Каждый сгенерированный ассет должен проходить внутренние критерии:

1. **Силуэт** — объект узнаваем без цвета.
2. **Простота** — минимум форм, без визуального шума.
3. **Целостность** — нет случайных деталей и «артефактов ИИ».
4. **Консистентность** — выглядит как часть одного набора.
5. **Игровая полезность** — подходит как юнит/проп для top-down/management игры.
6. **Техническая чистота SVG** — валидная разметка, прозрачный фон, разумная структура групп/слоёв.

Рекомендуемая структура SVG:

- корневой `viewBox` фиксированного логического размера (например `0 0 128 128` или `0 0 64 64`);
- группы по частям объекта (`body`, `head`, `legs`, `details`);
- плоские `fill` / умеренный `stroke`;
- без внешних зависимостей и растровых вставок.

---

## 10. Логика программы (что должно быть «внутри»)

Чтобы качество не зависело от удачи промпта, внутри продукта нужны собственные подсистемы:

### 10.1. Парсер запроса
Извлекает:
- субъект (`собака`);
- опциональные модификаторы (`рыжая`, `в броне`, `маленькая`);
- класс ассета.

### 10.2. Каталог визуальных примитивов / шаблонов
Для классов объектов заранее описаны простые схемы сборки:
- человек = туловище + голова + ноги (+ аксессуар роли);
- четвероногое = корпус + голова + 4 ноги + хвост;
- бочка = цилиндр/овал + обручи;
- и т.д.

### 10.3. Style Lock
Жёстко зафиксированный профиль стиля RimWorld / Prison Architect:
- ракурс;
- толщина линий;
- уровень детализации;
- палитра;
- запрещённые приёмы.

### 10.4. Генератор композиции
Собирает объект из простых частей по шаблону, а не «рисует всё сразу хаотично».

### 10.5. SVG Composer
Переводит структуру объекта в чистый SVG-код.

### 10.6. QC / проверка качества
Автоматические проверки:
- объект не пустой;
- не вылезает за `viewBox`;
- не слишком детализирован;
- есть базовые части для своего класса;
- файл валиден.

---

## 11. UX-требования

- Один главный экран: поле ввода + кнопка генерации + превью + download.
- Минимум настроек на старте.
- Не показывать пользователю сложные art-параметры.
- Ошибки формулировать понятно («не удалось распознать объект», «генерация не прошла проверку качества»).
- Показывать имя выходного файла и краткий тип ассета.

Опционально позже:
- история генераций;
- повторная генерация вариации;
- выбор размера `viewBox`;
- теги/категории.

---

## 12. Технические рамки первой версии (MVP)

MVP должен уметь:

1. Принять текстовый запрос.
2. Определить класс объекта.
3. Сгенерировать SVG в встроенном стиле.
4. Показать превью.
5. Сохранить файл.

Минимальный набор для проверки ценности:

- 5–10 персонажей;
- 5–10 животных;
- 5–10 монстров;
- 10–15 объектов мира.

Если на этом наборе стиль стабилен и объекты узнаваемы — архитектура верная.

---

## 13. Ограничения MVP (осознанно не делаем сразу)

- Полноценный редактор узлов SVG вручную.
- Сложные анимации и скелетная риггинг-система.
- Фотореализм / pixel-art / hand-painted HD.
- Генерация целых карт и тайловых наборов (это отдельный продукт/модуль).
- Обязательная интеграция с Unity/Godot в первой версии.
- Многопользовательский облачный сервис.

---

## 14. Критерии успеха

Проект достигает цели, если:

- пользователь может создать полезный игровой ассет за один короткий запрос;
- стиль RimWorld / Prison Architect соблюдается без ручных инструкций;
- результат — чистый SVG;
- качество стабильно на разных типах контента;
- система масштабируется от «собаки» к любым простым образам игрового мира.

---

## 15. Формулировка продуктового обещания

> Введи, что нужно для игры — получи простой, узнаваемый SVG-ассет в едином минималистичном стиле colony-sim / management игр. Без художника. Без объяснения стиля. Без лишней сложности.

---

## 16. Следующий шаг после этого документа

После утверждения дизайн-документа:

1. Уточнить технический стек (CLI / desktop / web).
2. Описать архитектуру модулей.
3. Зафиксировать style guide на примерах.
4. Спроектировать схему данных объекта и SVG-шаблоны.
5. Собрать MVP генерации «запрос → SVG».
